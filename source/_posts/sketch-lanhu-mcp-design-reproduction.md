最近尝试了一种新的前端还原方式：把一个静态专题页拆成“需求、原型、设计、代码”四个阶段。先根据需求和竞品用 Codex 生成原型，再把调整后的结构和文案交给 UI 做设计，等 UI 设计完成后，再用 MCP 辅助 AI 编码还原设计稿。

一开始我试的是 Sketch MCP，后来又用蓝湖 MCP 走了一次，发现它也可以完成类似的事情。这个流程的价值不在于“让 AI 直接替代 UI 或前端”，而在于把需求、原型、设计稿和代码串成一个相对连续的工作流，减少每个交接环节里的信息损耗。

## 起因

起点不是设计稿，而是一个静态专题页需求。

一开始我提供了需求和竞品，让 Codex 先生成一个静态专题页原型。这个方式能很快得到一个可讨论的页面，但初始结果的问题也很明显：页面结构比较散，信息层级不够清楚，文案和模块之间的关系也没有完全梳理好。

所以我没有直接把第一版当成可交付页面，而是基于它继续调整结构和文案。等页面逻辑、模块顺序、重点表达都相对清楚后，再把这部分内容交给 UI 做视觉设计。

UI 完成设计后，新的问题出现了：怎么把设计稿高效还原成代码。这个阶段如果只把截图交给 AI，它很容易又回到“凭视觉猜布局”的状态。于是我开始尝试用 MCP 读取设计稿里的结构、样式和标注，再让 AI 按这些信息复现页面。

## 处理流程

这个流程可以按四个阶段理解。

第一阶段是需求。这里先明确专题页要解决什么问题、面向谁、参考哪些竞品、需要表达哪些核心卖点。

第二阶段是原型。用 Codex 根据需求和竞品生成静态页面初稿，再人工整理结构和文案。这个阶段不追求视觉完成度，重点是把模块顺序、信息层级和表达重点梳理清楚。

第三阶段是设计。把整理后的原型和文案交给 UI，由设计师负责视觉层级、版式、组件风格和整体表达，形成正式设计稿。

第四阶段是代码。通过 MCP 读取画板、图层、组件层级、样式、标注和切图资源，再让 AI 按设计结构编码还原。生成后还要实际打开页面，对照设计稿检查：位置是否偏移、文字是否溢出、颜色是否明显不对、组件状态是否缺失。

## 为什么不用截图直接做

截图适合做快速理解，但不适合作为唯一依据。

截图最大的问题是它把所有东西压成了一张图。比如一个按钮的真实高度、内边距、文字样式、hover 状态、组件命名，在截图里都丢失了。AI 只能靠视觉猜测，结果经常是“看起来差不多”，但一进真实页面就容易露馅。

MCP 的好处是把设计稿重新拆回结构化信息。它能让 AI 知道这个区域不是一堆像素，而是一个容器、几段文字、一个按钮、一组图标和一套间距规则。

这会改变提示词的重点。以前是：

```text
请根据这张截图复现页面。
```

现在更像是：

```text
请根据这个画板的图层结构、样式标注和切图资源实现页面。
保持设计稿中的间距、字号、色值和层级关系。
```

后者给 AI 的约束更清楚，结果也更稳定。

## 两种 MCP 接入方式

Sketch MCP 和蓝湖 MCP 不是两个分工明确的角色，而是两种接入设计稿的路径。

Sketch MCP 的前提是能打开 Sketch 源文件。它的安装方式比较直接：先在 Sketch 里启用 MCP Server，然后编辑 Codex 的配置文件 `~/.codex/config.toml`，在里面新增一段 MCP server 配置：

```toml
[mcp_servers.sketch]
url = "http://localhost:31126/mcp"
```

这个方式适合设计师能提供源文件，或者设计文件本来就在本机可以打开的情况。它拿到的是设计源文件里的画板、图层、组件、样式和布局关系。

蓝湖 MCP 是后续尝试中发现的另一条路径。它的好处是：不需要设计师额外提供 Sketch 源文件，只要设计稿已经在蓝湖里，研发侧有项目链接和访问权限，就可以读取页面、标注、切图和设计分析结果。

蓝湖 MCP 是一个 GitHub 开源项目，地址是 `https://github.com/dsphper/lanhu-mcp`。它的 README 里给出的推荐方式很直接：让 AI 帮你克隆并安装这个项目。手动安装则有两条路线。

如果用 Docker，流程是：

```bash
git clone https://github.com/dsphper/lanhu-mcp.git
cd lanhu-mcp
bash setup-env.sh
docker-compose up -d
```

其中 `setup-env.sh` 会引导配置蓝湖 Cookie，并生成 `.env` 文件。

如果用源码运行，前置要求是 Python 3.10+，可以执行：

```bash
git clone https://github.com/dsphper/lanhu-mcp.git
cd lanhu-mcp
bash easy-install.sh
```

`easy-install.sh` 会安装依赖、引导获取 Cookie，并配置运行环境。服务启动后，默认 MCP 地址是 `http://localhost:8000/mcp`。之后同样编辑 `~/.codex/config.toml`，新增蓝湖 MCP 的配置：

```toml
[mcp_servers.lanhu]
url = "http://localhost:8000/mcp?role=Developer&name=Thomas"
```

实际使用时，蓝湖 MCP 不是去“浏览整个工作区”，而是围绕一个具体的蓝湖项目链接、邀请链接或设计稿链接工作。这个限制反而比较符合真实协作：研发通常拿到的也是某个项目或页面的交付链接。

## 一次验证

在后续尝试里，我用蓝湖 MCP 读取了一个蓝湖项目，项目里有 61 张设计图。选中“首页”之后，MCP 返回了页面规格、设计分析结果和切图资源，AI 再根据这些信息生成静态页面。

这次验证里，页面尺寸、主要结构和资源链路都能跑通，最终生成的页面也可以在本地打开并截图对比。初版仍然有一些明显问题，比如部分图标不对、响应式宽度还要继续调整，但整体流程是顺畅的：设计稿信息可以被读取，资源可以被拿到，代码可以生成，剩下的问题主要是视觉校准和细节修正。

这也是我觉得蓝湖 MCP 值得继续试的原因。它把“必须拿到设计源文件”这个前置条件去掉了，更接近很多团队真实的设计交付方式。

## 当前限制

这个流程还不是完全自动化。

首先，设计稿本身要比较规范。如果图层命名混乱、组件没有复用、视觉元素全是临时拼出来的，那么 MCP 读出来的数据也会比较乱。

其次，AI 对布局的判断仍然需要人工验收。即使有结构化数据，它也可能误解某些图层关系，比如把背景层当内容层，把装饰元素当交互控件，或者忽略响应式状态。

最后，资源链路也需要处理好。设计稿里的图片、图标、字体，如果没有正确导出或替换，页面仍然会和设计稿有差距。

所以这套流程更适合“快速生成第一版高还原页面”，不适合直接无检查上线。

## 短期结论

Sketch MCP 和蓝湖 MCP 的价值，不是让 AI 看图写页面，而是让 AI 读懂设计稿背后的结构和交付信息。

短期来看，我会把它当成一个提高初版还原速度的工具：先用 MCP 拉取结构、样式和资源，再让 AI 生成页面，最后人工做视觉验收和细节修正。

Sketch MCP 更适合有源文件的场景，蓝湖 MCP 更适合只有设计交付链接的场景。对我来说，后者的意义更大，因为它不要求设计师额外提供源文件，接入门槛更低，也更接近真实团队里的交付链路。

这比单纯截图复现稳定得多，也更符合真实前端开发流程。它不会消灭需求整理、UI 设计和前端验收里的人工判断，但能把大量低价值的抄尺寸、对颜色、查标注工作前移给工具处理。
